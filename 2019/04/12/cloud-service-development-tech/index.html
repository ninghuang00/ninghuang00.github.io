开源协议介绍

  1. GPL
    1. 软件始终开源,是否发布可执行文件随便,但是源代码必须发布
    2. 修改者可以去除原著中的版本信息,只要修改者的源代码遵守GPL开源协议
    3. 无论软件以什么形式发布,都必须同时附上源代码,修改者必须提供修改或者增加的源代码
    4. 开发或者维护遵循GPL协议软件的个人或者公司,可以向使用者收取使用费用
      修改代码之后必须提供源代码
  2. LGPL
    为类库使用设计的开原协议.使用LGPL协议的开源代码可以被商业软件作为类库引用,并发布和销售.但是不用开源商业软件的代码.但是如果对于引用的开源类库代码有修改,还是需要遵循LGPL协议开源.

基于开源技术的互联网架构

  1. 传统架构思维
    一个人不行,就找一个更强的人,减少中间沟通成本,单点故障明显,替换成本高,替换困难
  2. 互联网架构思维
    一个人不行,就找两个人,增加中间沟通成本,无单点,替换成本低,可以新增很多缓存区来提高系统整体性能

互联网架产品选型的关键要素

  • 性能：CDN、反向代理、应用服务、缓存、数据库、动静分离、图片分离
  • 可用性：服务器集群、缓存服务器集群、数据库分离、分表、主从分离
  • 伸缩性：遇到性能瓶颈时可以随时扩展、压力不大时可以随时减少
  • 扩展性：快速响应需求变化、时间驱动架构和分布式服务
  • 安全性：安全保护网站不受恶意访问和攻击、保护网站重要数据不被窃取

开源的分布式数据库缓存产品

商业数据库和开源数据库的区别

  1. 商业数据库
  • 优点
    1. 稳定性和可靠性高
    2. 强一致性要求高
    3. OLTP和OLAP处理能力强
    4. 技术文档和案例多
  • 缺点
    1. 可扩展性差，费用高
    2. 并发性能大数据处理能力差
  1. 开源数据库
  • 优点
    1. 高性能，高可用的分布式架构
    2. 灵活多样的数据类型
    3. 横向扩展性能好可以免费使用
    4. 非关系型数据库案例多
  • 缺点
    1. 技术支持厂家少
    2. 后期运维成本较高

开源分布式缓存软件

  1. Redis
    开源的高性能key-value数据库，基于内存也可以持久化

  2. Memcache
    自由开源的高性能、分布式的分布式内存对象缓存系统，通过在内存中维护一张巨大的hash表，能够存储各种格式的数据，包括图像、视频、文件以及数据库检索结果等。

  3. 选型对比

功能 Redis Memcache
CPU多核支持 单核处理 多核处理器
数据持久化 支持 不支持
宕机后数据找回 可通过AOF找回 数据丢失
是否支持复制功能 内置复制功能 需要第三方支持
K/V数据库缓存 支持 支持
数据备份 支持 不支持
应用场景 有持久化需求、对数据处理有高级要求的应用 简单的K/V存储
推荐应用场景 复杂场景应用 简单缓存场景使用，速度会比Redis快
